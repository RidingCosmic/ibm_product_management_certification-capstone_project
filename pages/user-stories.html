User Story Document

Solar-R-Us (SRU) LLC — Commercial & Residential App and Console. Part I: user stories by role, need, and value, each with acceptance criteria.

11 stories 6 epics Plan Phase
In my own words This part took longer than I expected. It's easy to write a story. It's harder to write acceptance criteria that actually mean something and aren't just restating the story back to you.

Stories below are organized by functional epic and sourced from the documented commercial and residential app/console requirements gathered during focus group sessions with management and prospective buyers.

Epic 1

Real-Time System Monitoring (Commercial App)

US-01

As a commercial customer, I want to monitor system efficiency in real time through the app, so that I can confirm the system is performing as expected and identify issues early.

Acceptance Criteria
  • App displays live efficiency metrics for the installed solar water heating system.
  • Data refreshes in real time, not on a manual-refresh-only basis.
  • Metrics are viewable on both Apple and Android devices.
  • User must have an installed system linked to the app to access this feature.
US-02

As a commercial customer, I want to receive real-time fault indicators when something goes wrong with my system, so that I can respond quickly and avoid extended downtime or service disruption.

Acceptance Criteria
  • App pushes a fault notification within real-time monitoring windows when an issue is detected.
  • Fault indicator includes a description of the issue type.
  • Indicator links directly to troubleshooting and diagnostic reports.
  • False-fault rate does not undermine user trust in the notification (no excessive false alerts).
US-03

As a commercial customer, I want to access diagnostic reports and automated fault eradication through the app, so that I can resolve common issues without waiting for a technician visit.

Acceptance Criteria
  • Diagnostic report is generated automatically when a fault is flagged.
  • App attempts automated fault eradication where the issue is software-resolvable.
  • If automated resolution fails, the app provides clear next-step guidance.
  • User can view a history of past faults and resolutions.
US-04

As a commercial customer, I want to configure my system settings remotely through the app, so that I can adjust the system to meet changing environmental conditions without an on-site visit.

Acceptance Criteria
  • App allows remote adjustment of relevant system configuration parameters.
  • Configuration changes are confirmed back to the user once applied.
  • Invalid configuration attempts are rejected with a clear error message.
  • Configuration changes are logged for later reference.
Epic 2

Residential Console

US-05

As a residential customer, I want a standard console linked to my system to monitor and adjust operations and outputs, so that I can manage my home solar water heating system without needing the full mobile app.

Acceptance Criteria
  • Console displays current system status, output, and key operational metrics.
  • Console allows manual adjustment of system operations.
  • Console connects to the system wirelessly with no hard connections required.
  • Console is mobile and can be relocated within range of the system.
US-06

As a residential customer, I want the console to automatically detect optimal solar collection settings and adjust positioning accordingly, so that I get the best system performance without having to manually calculate optimal settings.

Acceptance Criteria
  • Console automatically detects current environmental/weather conditions relevant to solar collection.
  • Console adjusts collector positioning settings automatically based on detected conditions.
  • User can override automatic adjustments manually if desired.
  • User is notified when an automatic adjustment is made.
Epic 3

Support and Help Desk Access

US-07

As a commercial or residential customer, I want the app to link directly to a 24/7 help desk function, so that I can get help immediately when I run into a problem, regardless of time of day.

Acceptance Criteria
  • Help desk link is accessible from the main app screen at all times.
  • Help desk function is available 24 hours a day, 7 days a week, prior to product launch.
  • Connection method (chat, call, or ticket) is clearly presented to the user.
US-08

As a commercial or residential customer, I want a built-in troubleshooting guide and frequently asked questions section in the app, so that I can try to resolve simple issues myself before contacting support.

Acceptance Criteria
  • Troubleshooting guide is accessible from the app's help section.
  • FAQ covers common system and app issues identified during beta testing.
  • Guide content is searchable or organized by topic for quick lookup.
Epic 4

Data and Diagnostics for Product Management

US-09

As the SRU product management team, I want to track app users, issues, and diagnostic information, so that I can determine system upgrade requirements, manage sustainability costs, and fine-tune marketing efforts.

Acceptance Criteria
  • App captures anonymized usage data including active users and reported issues.
  • Diagnostic information from user systems is logged centrally for analysis.
  • Data is available in a reportable format for the product management team.
  • Data collection complies with user privacy and security requirements (see US-10).
Epic 5

Security and Privacy

US-10

As a commercial or residential customer, I want assurance that my information is secure and not shared with third parties, so that I can trust the app with my personal and system data.

Acceptance Criteria
  • App does not share user information with third parties without explicit consent.
  • User data is encrypted in transit and at rest.
  • App includes a clear, accessible privacy statement covering data use.
  • Security requirement applies equally to both commercial and residential app versions.
Epic 6

Cross-Platform Access

US-11

As a commercial or residential customer, I want an app available on both Apple and Android, free to download, so that I can use the app regardless of which mobile device I own, with no added cost.

Acceptance Criteria
  • App is published and functional on both the Apple App Store and Google Play.
  • App is free to download for all buyers and users.
  • User must have an installed system to access in-app features, consistent across both platforms.
  • Core functionality (monitoring, faults, configuration) is at parity across both platforms.
In my own words If I had to pick the most important story on this list, it's US-10. Security tends to get treated as a checkbox at the end, but for a product collecting real-time system data from someone's home or business, it has to be a first-class requirement, not an afterthought.
